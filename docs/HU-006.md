## Historia de Usuario

**Como** usuario del sistema,  
**quiero** que un agente de inteligencia artificial interprete mi pregunta en lenguaje natural, razone sobre los datos necesarios, llame al modelo de predicción con el payload correcto y formule una respuesta comprensible,  
**para** interactuar con el modelo predictivo sin conocer su interfaz técnica ni la estructura de los datos de entrada.

---

## Descripción

El agente implementa el patrón ReAct (Reasoning + Acting), utilizando LiteLLM con Groq como proveedor LLM (modelo `llama3-8b-8192`). El agente sigue un ciclo explícito de pensamiento-acción-observación: primero razona sobre la pregunta para identificar provincia y mes, luego actúa consultando datos históricos del CSV para construir el payload completo, luego actúa llamando al servicio de inferencia, y finalmente razona sobre el resultado para formular una respuesta en lenguaje natural contextualizada para el dominio de ventas del Ecuador.

---

## Criterios de Aceptación

### Escenario 1: Extracción correcta de entidades desde lenguaje natural

**Given** que el agente recibe la pregunta "¿Cuánto venderán las sociedades de Pichincha en octubre?"  
**When** ejecuta su fase de razonamiento inicial  
**Then** el log del agente muestra `[THINK] Provincia identificada: PICHINCHA, Mes identificado: 10`  
**And** las entidades extraídas coinciden con los valores esperados  
**And** si no puede extraer alguna entidad, el log muestra `[THINK] Información insuficiente: falta <entidad>`  

### Escenario 2: Construcción correcta del payload de inferencia

**Given** que el agente ha identificado provincia GUAYAS y mes 7  
**When** ejecuta la tool `get_province_data`  
**Then** el log muestra `[ACT] Consultando datos históricos para GUAYAS, mes 7`  
**And** el agente recupera del CSV los valores más recientes disponibles de las features requeridas para esa provincia  
**And** el log muestra `[OBSERVE] Datos recuperados: {exportaciones_bienes_pn: X, ...}`  
**And** construye el payload completo para el endpoint `/predict`  

### Escenario 3: Llamada exitosa al servicio de inferencia

**Given** que el agente tiene el payload construido  
**When** ejecuta la tool `call_inference`  
**Then** el log muestra `[ACT] Llamando a POST /predict con payload: {...}`  
**And** recibe la respuesta del modelo con la predicción y la versión del modelo  
**And** el log muestra `[OBSERVE] Predicción recibida: $X (modelo v2, confianza: 0.87)`  

### Escenario 4: Formulación de respuesta en lenguaje natural

**Given** que el agente ha recibido la predicción del modelo  
**When** ejecuta su fase de razonamiento final  
**Then** genera una respuesta en español que incluye:
- El valor predicho formateado como moneda (USD)
- La provincia y mes en contexto
- Una breve interpretación del resultado en términos de negocio
- La versión del modelo utilizado
**And** el log muestra `[THINK] Formulando respuesta final con predicción $X`  

### Escenario 5: El servicio de inferencia no está disponible

**Given** que el servicio `ml-inference` está caído  
**When** el agente intenta llamar a `/predict`  
**Then** el log muestra `[OBSERVE] Error al llamar inferencia: Connection refused`  
**And** el agente retorna una respuesta al usuario indicando que el servicio no está disponible temporalmente  
**And** no se lanza una excepción no manejada

## Definición de Done

- [ ] `agent.py` implementa la clase `ReActAgent` con el ciclo Thought → Action → Observation
- [ ] `tools.py` define al menos dos tools: `get_province_data` y `call_inference`
- [ ] Cada paso del ciclo ReAct genera una entrada en el log con prefijo `[THINK]`, `[ACT]` u `[OBSERVE]`
- [ ] El agente usa LiteLLM con configuración `model="groq/llama3-8b-8192"`
- [ ] La API key de Groq se inyecta via variable de entorno `GROQ_API_KEY`, nunca hardcodeada
- [ ] La URL del servicio de inferencia se configura via variable de entorno `INFERENCE_SERVICE_URL`
- [ ] El endpoint `POST /process` retorna el campo `razonamiento` con el log del ciclo ReAct
- [ ] El agente publica el evento en Kafka topic `agent-actions` al finalizar su ciclo
- [ ] El tiempo de respuesta del agente se expone como métrica Prometheus `agent_processing_duration_seconds`
- [ ] Tests unitarios validan la extracción de entidades con al menos 5 preguntas de ejemplo
- [ ] Tests unitarios usan mocks de LiteLLM y del servicio de inferencia
- [ ] El Dockerfile construye sin errores y el servicio arranca correctamente
