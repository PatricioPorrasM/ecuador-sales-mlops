---

## Historia de Usuario

**Como** ingeniero de ML del equipo,  
**quiero** que el entrenamiento de los modelos se dispare automáticamente cuando se detecta un cambio en el código del trainer o en los datos de entrenamiento al hacer push a la rama principal,  
**para** garantizar que el modelo en producción siempre refleje la última versión validada del pipeline de entrenamiento sin intervención manual.

---

## Descripción

El pipeline de CI/CD debe detectar cambios en el directorio `services/model-trainer/` y ejecutar automáticamente un Kubernetes Job que entrena dos versiones del modelo (v1 con RandomForestRegressor y v2 con XGBoostRegressor) usando el dataset histórico de ventas provinciales del SRI del Ecuador. Cada ejecución debe quedar registrada como un experimento en Weights & Biases con todas sus métricas e hiperparámetros.

---

## Criterios de Aceptación

### Escenario 1: Push a main dispara entrenamiento automático

**Given** que un desarrollador hace push a la rama `main` con cambios en `services/model-trainer/`  
**When** el workflow de GitHub Actions se activa  
**Then** se ejecuta el job de entrenamiento en el entorno de CI  
**And** se entrena el modelo v1 (RandomForest) y el modelo v2 (XGBoost) de forma secuencial  
**And** ambos runs quedan registrados en el proyecto de Weights & Biases configurado  
**And** el workflow falla con código de error distinto de cero si algún entrenamiento lanza una excepción  

### Escenario 2: Métricas registradas correctamente en W&B

**Given** que el job de entrenamiento se ejecuta exitosamente  
**When** finaliza el entrenamiento de cada versión del modelo  
**Then** en el proyecto de W&B aparece un nuevo run con el nombre `sales-v1-<timestamp>` o `sales-v2-<timestamp>`  
**And** el run registra las métricas: RMSE, MAE, MAPE y R²  
**And** el run registra los hiperparámetros utilizados  
**And** el run registra las curvas de aprendizaje (train vs validation loss por epoch o estimador)  
**And** el modelo entrenado queda guardado como artifact en W&B  

### Escenario 3: El dataset es cargado correctamente desde el volumen

**Given** que el archivo `sales_data.csv` existe en la ruta configurada  
**When** el trainer inicia su ejecución  
**Then** carga el CSV sin errores  
**And** aplica el preprocesamiento definido (features de lag, rolling mean, codificación cíclica del mes para v2)  
**And** divide los datos en train/validation respetando el orden temporal (sin data leakage)  

### Escenario 4: Fallo controlado si el dataset no existe

**Given** que el archivo `sales_data.csv` no existe en la ruta esperada  
**When** el trainer intenta cargarlo  
**Then** el proceso termina con un mensaje de error claro en los logs  
**And** el run en W&B queda marcado como fallido  
**And** el Kubernetes Job refleja estado `Failed`  

---

## Definición de Done

- [ ] `train_v1.py` entrena RandomForestRegressor y registra run completo en W&B
- [ ] `train_v2.py` entrena XGBoostRegressor con features enriquecidas y registra run completo en W&B
- [ ] La división train/validation respeta el orden temporal (sin shuffle)
- [ ] Las métricas RMSE, MAE, MAPE y R² se calculan sobre el conjunto de validación
- [ ] Los modelos entrenados se serializan con joblib y se guardan como artifacts en W&B
- [ ] El Kubernetes Job `model-trainer` ejecuta ambos entrenamientos en secuencia
- [ ] El GitHub Actions workflow `cd.yml` incluye el step de trigger del Job de entrenamiento
- [ ] El Dockerfile del trainer construye sin errores con python:3.11-slim
- [ ] Los logs del entrenamiento son estructurados e incluyen timestamps
- [ ] El `requirements.txt` del trainer está completo y con versiones fijadas
- [ ] Tests unitarios validan el preprocesamiento de features con datos sintéticos
