---

## Historia de Usuario

**Como** usuario del sistema o auditor técnico,  
**quiero** que cada predicción generada por el modelo incluya información sobre qué versión del modelo la produjo y un indicador de confianza del resultado,  
**para** evaluar la fiabilidad de la predicción, saber si el modelo ha sido actualizado recientemente y poder tomar decisiones informadas sobre el grado de certeza del valor predicho.

---

## Descripción

El servicio de inferencia debe retornar junto con cada predicción: la versión del modelo que la generó (v1 o v2), un score de confianza calculado a partir de la dispersión de los árboles del ensemble (desviación estándar normalizada entre predicciones individuales de los estimadores), y metadatos básicos de la request. Esta información se muestra en el panel colapsable de la interfaz de chat y se registra en las métricas de Prometheus y en el evento de Kafka.

---

## Criterios de Aceptación

### Escenario 1: Respuesta del endpoint /predict incluye versión y confianza

**Given** que el servicio `ml-inference` está corriendo con un modelo cargado  
**When** se hace una petición `POST /predict` con un payload válido  
**Then** la respuesta HTTP 200 incluye el siguiente esquema:
```json
{
  "provincia": "GUAYAS",
  "mes": 7,
  "prediccion_total_ventas": 4567890123.45,
  "modelo_version": "v2",
  "confianza": 0.87,
  "prediccion_formateada": "$4,567,890,123.45"
}
```
**And** el campo `modelo_version` corresponde a la versión del modelo actualmente en producción  
**And** el campo `confianza` es un valor entre 0.0 y 1.0  

### Escenario 2: El score de confianza refleja la dispersión del ensemble

**Given** que el modelo cargado es un ensemble (RandomForest o XGBoost)  
**When** se procesa una predicción  
**Then** el score de confianza se calcula como `1 - (std_predicciones / mean_predicciones)` normalizado entre 0 y 1  
**And** predicciones con alta dispersión entre estimadores producen un score de confianza bajo  
**And** predicciones con baja dispersión producen un score de confianza alto  

### Escenario 3: La versión del modelo se expone como métrica Prometheus

**Given** que el servicio `ml-inference` tiene un modelo cargado  
**When** Prometheus hace scraping del endpoint `/metrics`  
**Then** la métrica `model_version_info` está presente con el label `version` indicando `v1` o `v2`  
**And** la métrica `model_confidence_score` acumula el histograma de scores de confianza de todas las predicciones  

### Escenario 4: La interfaz de chat muestra la información de versión y confianza

**Given** que el agente ha recibido la respuesta del servicio de inferencia  
**When** la respuesta es mostrada en el chat  
**Then** el panel colapsable "Datos técnicos" incluye:
- Versión del modelo: v1 o v2
- Score de confianza: valor porcentual (ej. 87%)
- Provincia detectada y mes detectado
- Valor predicho sin formato y formateado  

### Escenario 5: El evento Kafka incluye versión y confianza

**Given** que una predicción ha sido generada exitosamente  
**When** el evento se publica en el topic `model-responses`  
**Then** el mensaje incluye los campos `modelo_version` y `confianza`  
**And** el `kafka-consumer` los utiliza para calcular métricas de confianza promedio por provincia

## Definición de Done

- [ ] `model_loader.py` expone el método `predict_with_metadata()` que retorna predicción, versión y confianza
- [ ] El cálculo de confianza usa `estimators_` del RandomForest o `get_booster()` de XGBoost
- [ ] El endpoint `POST /predict` retorna siempre los campos `modelo_version`, `confianza` y `prediccion_formateada`
- [ ] La métrica Prometheus `model_version_info` se actualiza al cargar el modelo
- [ ] La métrica Prometheus `model_confidence_score` registra el histograma de scores
- [ ] La interfaz de chat muestra el panel colapsable con los datos técnicos
- [ ] El evento Kafka `model-responses` incluye `modelo_version` y `confianza`
- [ ] Tests unitarios validan que el score de confianza está entre 0.0 y 1.0
- [ ] Tests unitarios validan el formato correcto de la respuesta del endpoint
