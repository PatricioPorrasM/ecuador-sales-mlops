# HU-008: Integración Continua con Validación de Calidad

## Identificación

| Campo | Detalle |
|---|---|
| **ID** | HU-008 |
| **Épica** | CI/CD |
| **Prioridad** | Media-Alta |
| **Estimación** | 5 puntos |
| **Sprint** | 3 |

---

## Historia de Usuario

**Como** desarrollador del equipo,  
**quiero** que cada push al repositorio ejecute automáticamente un pipeline de integración continua que valide la calidad del código, ejecute los tests y escanee las imágenes Docker en busca de vulnerabilidades,  
**para** detectar errores, regresiones y problemas de seguridad antes de que el código llegue a producción, manteniendo la calidad del sistema de forma automatizada.

---

## Descripción

El workflow de CI en GitHub Actions se activa con cada push a cualquier rama y con cada Pull Request hacia `main`. El pipeline ejecuta cuatro etapas en secuencia: análisis estático del código con flake8, ejecución de tests unitarios con pytest y reporte de cobertura, escaneo de seguridad de las imágenes Docker con Trivy, y construcción de las imágenes para validar que los Dockerfiles son correctos. El pipeline de CD solo se activa cuando el CI completa exitosamente en la rama `main`.

---

## Criterios de Aceptación

### Escenario 1: Push a cualquier rama activa el pipeline de CI

**Given** que un desarrollador hace push a cualquier rama del repositorio  
**When** el workflow `ci.yml` se activa  
**Then** se ejecutan los jobs: `lint`, `test`, `security-scan` y `build` en ese orden  
**And** si algún job falla, los jobs siguientes no se ejecutan  
**And** el resultado del pipeline es visible en la pestaña Actions del repositorio  

### Escenario 2: El job de lint detecta errores de estilo

**Given** que el código de algún servicio tiene una violación de estilo PEP8  
**When** el job `lint` ejecuta `flake8` sobre todos los servicios  
**Then** el job falla con código de salida distinto de cero  
**And** el log muestra el archivo, línea y descripción de la violación  
**And** los jobs siguientes (`test`, `security-scan`, `build`) no se ejecutan  

### Escenario 3: El job de test ejecuta pruebas con reporte de cobertura

**Given** que todos los tests están correctamente definidos en los directorios `tests/` de cada servicio  
**When** el job `test` ejecuta `pytest --cov`  
**Then** todos los tests pasan exitosamente  
**And** el reporte de cobertura muestra un porcentaje mínimo del 70% para cada servicio  
**And** si la cobertura es menor al 70% en algún servicio, el job falla  
**And** el reporte de cobertura se sube como artifact del workflow  

### Escenario 4: El job de security-scan detecta vulnerabilidades críticas

**Given** que las imágenes Docker han sido construidas  
**When** el job `security-scan` ejecuta Trivy sobre cada imagen  
**Then** se genera un reporte de vulnerabilidades para cada imagen  
**And** si se detectan vulnerabilidades de severidad CRITICAL, el job falla  
**And** el reporte se sube como artifact del workflow  

### Escenario 5: El CD se activa solo cuando CI completa en main

**Given** que un PR ha sido aprobado y mergeado a `main`  
**When** el workflow `ci.yml` completa exitosamente  
**Then** el workflow `cd.yml` se activa automáticamente  
**And** `cd.yml` construye y publica las imágenes Docker a GitHub Container Registry con el tag del SHA del commit  
**And** aplica los manifiestos Kubernetes actualizados  
**And** ejecuta un smoke test verificando que `/health` de cada servicio retorna HTTP 200  

### Escenario 6: Smoke test post-deploy valida el sistema

**Given** que el deploy en Kubernetes ha completado  
**When** el job `smoke-test` del workflow `cd.yml` se ejecuta  
**Then** realiza un `curl` al endpoint `/health` de `web-ui`, `ai-agent` y `ml-inference`  
**And** verifica que cada uno retorna HTTP 200  
**And** si algún servicio no responde, el job falla y notifica que el deploy requiere revisión

## Definición de Done

- [ ] `.github/workflows/ci.yml` con jobs: `lint`, `test`, `security-scan`, `build`
- [ ] `.github/workflows/cd.yml` con jobs: `build-and-push`, `deploy`, `smoke-test`
- [ ] `cd.yml` tiene `needs: ci` y solo se activa en pushes a `main`
- [ ] flake8 configurado con máximo de línea de 100 caracteres (`--max-line-length=100`)
- [ ] pytest configurado con `--cov` y umbral mínimo de 70% (`--cov-fail-under=70`)
- [ ] Trivy configurado para fallar solo en vulnerabilidades CRITICAL (`--severity CRITICAL`)
- [ ] Las imágenes se publican a `ghcr.io/<owner>/<repo>/<service>:<sha>`
- [ ] Los Secrets de GitHub configurados: `GROQ_API_KEY`, `WANDB_API_KEY`, `KUBECONFIG`
- [ ] Los reportes de cobertura y seguridad se suben como artifacts del workflow
- [ ] El README incluye badge del estado del pipeline de CI
- [ ] La documentación describe cómo configurar los Secrets en GitHub
