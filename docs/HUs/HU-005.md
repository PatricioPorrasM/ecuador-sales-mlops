## Historia de Usuario

**Como** ingeniero de plataforma o stakeholder técnico del proyecto,  
**quiero** visualizar en un dashboard de Grafana el estado en tiempo real del sistema y del modelo,  
**para** detectar degradación de rendimiento, cuellos de botella, errores y comportamientos anómalos sin necesidad de revisar logs manualmente.

---

## Descripción

Prometheus recolecta métricas expuestas por cada servicio Flask a través de endpoints `/metrics`. Grafana consume Prometheus como fuente de datos y presenta un dashboard unificado con paneles que cubren tanto métricas de infraestructura (latencia, errores, CPU, memoria) como métricas de negocio y modelo (distribución de predicciones, consultas por provincia, confianza del modelo, lag de Kafka).

---

## Criterios de Aceptación

### Escenario 1: Prometheus recolecta métricas de todos los servicios

**Given** que todos los servicios están corriendo en Kubernetes  
**When** Prometheus ejecuta su ciclo de scraping configurado (cada 15 segundos)  
**Then** los targets de `web-ui`, `ai-agent`, `ml-inference` y `kafka-consumer` aparecen como `UP` en la UI de Prometheus  
**And** las métricas están disponibles para consulta en PromQL  

### Escenario 2: Dashboard de Grafana muestra métricas de infraestructura

**Given** que Grafana está configurado con Prometheus como datasource  
**When** se abre el dashboard principal  
**Then** son visibles los siguientes paneles con datos actualizados:
- Requests por minuto por servicio (últimos 30 minutos)
- Latencia P50, P95 y P99 del endpoint de inferencia
- Tasa de errores HTTP 4xx y 5xx por servicio
- Latencia end-to-end del pipeline completo (P50, P95)

### Escenario 3: Dashboard muestra métricas del modelo y negocio

**Given** que se han realizado al menos 5 consultas al sistema  
**When** se visualiza el dashboard en Grafana  
**Then** son visibles los siguientes paneles:
- Distribución del valor de las predicciones (histograma)
- Top 5 provincias más consultadas (bar chart)
- Score de confianza promedio del modelo (gauge)
- Versión del modelo actualmente en producción (stat panel)

### Escenario 4: Dashboard muestra métricas de Kafka

**Given** que el `kafka-consumer` está corriendo  
**When** se visualiza el dashboard  
**Then** son visibles:
- Mensajes consumidos por topic (contador acumulado)
- Lag del consumer por topic (gauge)
- Tasa de mensajes por minuto por topic

### Escenario 5: Acceso al dashboard desde el exterior del cluster

**Given** que el sistema está desplegado en Minikube  
**When** el usuario ejecuta `minikube service grafana -n monitoring`  
**Then** el browser abre la UI de Grafana en el puerto NodePort 30300  
**And** el dashboard principal está precargado y visible sin configuración adicional

## Definición de Done

- [ ] `prometheus.yml` configura scraping de los 4 servicios con intervalo de 15s
- [ ] Cada servicio Flask expone `/metrics` con métricas Prometheus usando `prometheus-client`
- [ ] Las siguientes métricas están implementadas en cada servicio:
  - `http_requests_total` (counter, labels: method, endpoint, status)
  - `http_request_duration_seconds` (histogram, labels: endpoint)
- [ ] El servicio `ml-inference` expone adicionalmente:
  - `model_prediction_value` (gauge)
  - `model_confidence_score` (histogram)
  - `model_version_info` (gauge con label version)
- [ ] El servicio `kafka-consumer` expone:
  - `kafka_messages_consumed_total` (counter, label: topic)
  - `pipeline_end_to_end_latency_seconds` (histogram)
  - `requests_by_provincia` (counter, label: provincia)
- [ ] El archivo `grafana/dashboards/main-dashboard.json` está versionado en el repo
- [ ] El dashboard se carga automáticamente al iniciar Grafana vía provisioning
- [ ] El datasource de Prometheus está configurado vía provisioning (no manual)
- [ ] El Service de Grafana en Kubernetes es NodePort en el puerto 30300
- [ ] El README incluye captura de pantalla del dashboard y descripción de cada panel
