# HU-009: Inferencia con Información de Versión y Confianza

## Identificación

| Campo | Detalle |
|---|---|
| **ID** | HU-009 |
| **Épica** | Servicio de Inferencia |
| **Prioridad** | Media |
| **Estimación** | 3 puntos |
| **Sprint** | 2 |
| **Versión** | 2.0 — incorpora `año` como campo obligatorio del payload de inferencia y como feature del modelo |

---

## Historia de Usuario

**Como** usuario del sistema o auditor técnico,
**quiero** que cada predicción generada incluya la información del período completo (mes **y año**), la versión del modelo que la produjo y un indicador de confianza del resultado,
**para** evaluar la fiabilidad de la predicción, saber con precisión para qué período se predijo, identificar qué versión del modelo respondió y tomar decisiones informadas sobre el grado de certeza del valor predicho.

---

## Descripción

El servicio de inferencia recibe como entrada el año solicitado (además de provincia, mes y las variables de exportación) y lo incluye como feature en la inferencia del modelo, ya que el año fiscal captura la tendencia inter-anual de las ventas en el dataset del SRI. La respuesta incluye el año del período predicho, la versión del modelo, un score de confianza calculado desde la dispersión del ensemble, y el valor formateado en dólares. Esta información se expone en el panel colapsable del chat, en las métricas de Prometheus y en el evento de Kafka.

---

## Criterios de Aceptación

### Escenario 1: El payload de entrada incluye el campo `año` como obligatorio

**Given** que el agente ha validado una fecha y construido el payload de inferencia
**When** llama al endpoint `POST /predict`
**And** el body incluye el campo `año`
**Then** el servicio acepta la request con HTTP 200
**And** si el campo `año` está ausente en el body, el servicio retorna HTTP 422 con un mensaje de validación claro: `{"error": "El campo 'año' es obligatorio"}`

### Escenario 2: El año es usado como feature en la inferencia del modelo

**Given** que el modelo fue entrenado con `año_fiscal` como feature
**When** el servicio recibe una request con `año: 2026`
**Then** el valor `2026` se incluye en el vector de features antes de llamar al modelo
**And** el log del servicio indica: `[INFERENCE] Features: {..., año_fiscal: 2026}`
**And** la predicción refleja la tendencia inter-anual aprendida durante el entrenamiento

### Escenario 3: La respuesta incluye el año del período predicho

**Given** que el servicio recibe y procesa una solicitud válida con `año: 2026`
**When** el modelo genera la predicción
**Then** la respuesta HTTP 200 incluye el siguiente esquema:
```json
{
  "provincia": "GUAYAS",
  "mes": 3,
  "año": 2026,
  "periodo": "marzo 2026",
  "prediccion_total_ventas": 4567890123.45,
  "prediccion_formateada": "$4,567,890,123.45",
  "modelo_version": "v2",
  "confianza": 0.87
}
```
**And** el campo `periodo` es una representación legible del mes en texto más el año

### Escenario 4: El score de confianza refleja la dispersión del ensemble

**Given** que el modelo cargado es un ensemble (RandomForest o XGBoost)
**When** se procesa una predicción
**Then** el score de confianza se calcula como `1 - (std_predicciones / mean_predicciones)` normalizado entre 0 y 1
**And** predicciones con alta dispersión entre estimadores producen un score bajo
**And** el campo `confianza` siempre está entre 0.0 y 1.0

### Escenario 5: La versión y confianza del modelo se exponen como métricas Prometheus

**Given** que el servicio `ml-inference` tiene un modelo cargado
**When** Prometheus hace scraping de `/metrics`
**Then** la métrica `model_version_info` está presente con el label `version` indicando `v1` o `v2`
**And** la métrica `model_confidence_score` acumula el histograma de scores de todas las predicciones
**And** la métrica `inference_requests_total` tiene label `año` para análisis por período

### Escenario 6: La interfaz de chat muestra la información completa de versión, confianza y período

**Given** que el agente recibió la respuesta del servicio de inferencia
**When** la respuesta es mostrada en el chat
**Then** el texto principal incluye el período en formato legible: *"para marzo de 2026"*
**And** el panel colapsable "Datos técnicos" incluye:
  - Período predicho: marzo 2026
  - Versión del modelo: v1 o v2
  - Score de confianza: valor porcentual (ej. 87%)
  - Valor predicho sin formato y formateado

### Escenario 7: El evento Kafka incluye año, versión y confianza

**Given** que una predicción ha sido generada exitosamente
**When** el evento se publica en el topic `model-responses`
**Then** el mensaje incluye los campos `año`, `modelo_version` y `confianza`
**And** el `kafka-consumer` usa el campo `año` para calcular la métrica `requests_by_año` (counter, label: año)

## Definición de Done

- [ ] El endpoint `POST /predict` requiere el campo `año` en el body y retorna HTTP 422 si está ausente
- [ ] El valor de `año` se incluye como feature `año_fiscal` en el vector de inferencia del modelo
- [ ] La respuesta del endpoint incluye los campos: `provincia`, `mes`, `año`, `periodo`, `prediccion_total_ventas`, `prediccion_formateada`, `modelo_version`, `confianza`
- [ ] El campo `periodo` se genera como texto en español (ej. "marzo 2026")
- [ ] `model_loader.py` expone `predict_with_metadata(provincia, mes, año, ...)` retornando predicción, versión y confianza
- [ ] El cálculo de confianza usa `estimators_` de RandomForest o predicciones individuales de árboles en XGBoost
- [ ] La métrica `model_version_info` se actualiza al cargar el modelo
- [ ] La métrica `model_confidence_score` registra el histograma de scores
- [ ] La métrica `inference_requests_total` tiene label `año`
- [ ] El evento Kafka `model-responses` incluye `año`, `modelo_version` y `confianza`
- [ ] La interfaz de chat muestra el período en texto y el panel colapsable con todos los campos técnicos
- [ ] Tests unitarios validan que el score de confianza está entre 0.0 y 1.0
- [ ] Tests unitarios validan que la ausencia del campo `año` retorna HTTP 422
- [ ] Tests unitarios validan el formato correcto del campo `periodo` para distintos meses y años
