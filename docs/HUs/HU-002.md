## Historia de Usuario

**Como** ingeniero de ML del equipo,
**quiero** que el entrenamiento de los modelos se dispare automáticamente cuando se detecta un cambio en el código del trainer o en los datos de entrenamiento al hacer push a la rama principal,
**para** garantizar que el modelo en producción siempre refleje la última versión validada del pipeline de entrenamiento, incluyendo la feature de año fiscal derivada correctamente del dataset del SRI.

---

## Descripción

El pipeline de CI/CD detecta cambios en `services/model-trainer/` y ejecuta un Kubernetes Job que entrena dos versiones del modelo usando el dataset histórico de ventas provinciales del SRI del Ecuador (enero 2020 – septiembre 2025). El dataset no contiene columna de año explícita: el año y mes fiscal se derivan del número de registro secuencial mediante una regla fija. Ambas versiones del modelo incluyen `año_fiscal` como feature, ya que captura la tendencia inter-anual de las ventas. Cada ejecución queda registrada como experimento en Weights & Biases.

---

## Regla de derivación del año y mes fiscal

El dataset tiene registros secuenciales numerados desde 1. La derivación es:

```
año_fiscal = 2020 + (numero_registro - 1) // 12
mes_fiscal  = ((numero_registro - 1) % 12) + 1
```

| Registro | Año fiscal | Mes fiscal |
|---|---|---|
| 1 | 2020 | 1 (enero) |
| 12 | 2020 | 12 (diciembre) |
| 13 | 2021 | 1 (enero) |
| 25 | 2022 | 1 (enero) |
| 69 | 2025 | 9 (septiembre) |

Esta lógica debe implementarse en el preprocesamiento del trainer y estar cubierta por tests unitarios con los valores de la tabla anterior.

## Criterios de Aceptación

### Escenario 1: Push a main dispara entrenamiento automático

**Given** que un desarrollador hace push a `main` con cambios en `services/model-trainer/`
**When** el workflow de GitHub Actions se activa
**Then** se ejecuta el job de entrenamiento en el entorno de CI
**And** se entrenan el modelo v1 (RandomForest) y el modelo v2 (XGBoost) de forma secuencial
**And** ambos runs quedan registrados en el proyecto de Weights & Biases configurado
**And** el workflow falla con código de salida distinto de cero si algún entrenamiento lanza una excepción

### Escenario 2: La derivación del año y mes fiscal es correcta

**Given** que el archivo `sales_data.csv` tiene registros numerados secuencialmente desde 1
**When** el preprocesamiento del trainer carga y transforma el dataset
**Then** la columna `año_fiscal` toma el valor 2020 para los registros 1 a 12
**And** la columna `año_fiscal` toma el valor 2021 para los registros 13 a 24
**And** la columna `año_fiscal` toma el valor 2025 para los registros 61 a 69
**And** la columna `mes_fiscal` toma el valor 1 para los registros 1, 13, 25, 37, 49, 61
**And** la columna `mes_fiscal` toma el valor 9 para el registro 69 (último del dataset)

### Escenario 3: `año_fiscal` es incluido como feature en ambos modelos

**Given** que el preprocesamiento ha derivado correctamente `año_fiscal` y `mes_fiscal`
**When** se construyen los conjuntos de features para v1 y v2
**Then** la feature `año_fiscal` está presente en el conjunto de features de ambas versiones
**And** W&B registra en el panel de hiperparámetros la lista completa de features usadas
**And** el feature importance del modelo incluye `año_fiscal` como una de las variables evaluadas

### Escenario 4: Métricas registradas correctamente en W&B

**Given** que el job de entrenamiento se ejecuta exitosamente
**When** finaliza el entrenamiento de cada versión
**Then** en W&B aparece un nuevo run con nombre `sales-v1-<timestamp>` o `sales-v2-<timestamp>`
**And** el run registra las métricas: RMSE, MAE, MAPE y R² calculadas sobre el conjunto de validación
**And** el run registra los hiperparámetros utilizados y la lista de features
**And** el modelo entrenado queda guardado como artifact en W&B

### Escenario 5: La división train/validation respeta el orden temporal

**Given** que el dataset tiene 69 registros ordenados cronológicamente
**When** el trainer divide los datos en train y validation
**Then** los registros de validación corresponden a los meses más recientes del dataset
**And** no se aplica shuffle antes de la división (sin data leakage temporal)
**And** el porcentaje de validación es configurable via variable de entorno `VALIDATION_SPLIT` (default: 0.2)

### Escenario 6: Fallo controlado si el dataset no existe o tiene estructura incorrecta

**Given** que el archivo `sales_data.csv` no existe o le falta la columna `MES FISCAL`
**When** el trainer intenta cargarlo y preprocesarlo
**Then** el proceso termina con un mensaje de error claro en los logs indicando el problema específico
**And** el run en W&B queda marcado como fallido
**And** el Kubernetes Job refleja estado `Failed`
