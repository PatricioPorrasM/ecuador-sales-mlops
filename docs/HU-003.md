---

## Historia de Usuario

**Como** ingeniero de ML del equipo,  
**quiero** que el sistema compare automáticamente las métricas de los modelos v1 y v2 después de cada entrenamiento y promueva al mejor al estado de producción en el W&B Model Registry,  
**para** garantizar que el servicio de inferencia siempre sirva el modelo con mejor desempeño demostrado, eliminando la decisión manual y el riesgo de regresión.

---

## Descripción

Después de que ambos modelos son entrenados y registrados en W&B, un script de comparación automatizada evalúa sus métricas de validación. Si el modelo v2 supera al v1 por un margen mínimo definido (reducción del RMSE mayor al 5%), v2 es promovido a estado `production` en el W&B Model Registry. En caso contrario, v1 es promovido. El modelo en estado `production` es el que el servicio de inferencia descarga y utiliza.

---

## Criterios de Aceptación

### Escenario 1: v2 supera a v1 por el margen mínimo definido

**Given** que ambos modelos han sido entrenados y sus runs están registrados en W&B  
**When** el script `compare_and_promote.py` se ejecuta  
**And** el RMSE del modelo v2 es menor al RMSE del modelo v1 multiplicado por 0.95  
**Then** el artifact del modelo v2 es promovido al estado `production` en el W&B Model Registry  
**And** el artifact del modelo v1 queda en estado `archived`  
**And** el log del script indica claramente: `[PROMOTION] Modelo v2 promovido a production. RMSE v1: X, RMSE v2: Y, mejora: Z%`  

### Escenario 2: v2 no supera a v1 por el margen mínimo

**Given** que ambos modelos han sido entrenados  
**When** el script de comparación se ejecuta  
**And** la mejora en RMSE de v2 sobre v1 es menor al 5%  
**Then** el artifact del modelo v1 es promovido al estado `production`  
**And** el log indica: `[PROMOTION] Modelo v1 promovido a production. La mejora de v2 (Z%) no supera el umbral mínimo del 5%`  

### Escenario 3: El servicio de inferencia carga el modelo en estado production

**Given** que existe un modelo con estado `production` en el W&B Model Registry  
**When** el pod de inferencia `ml-inference` arranca  
**Then** descarga automáticamente el artifact marcado como `production`  
**And** lo carga en memoria correctamente  
**And** el endpoint `GET /ready` retorna HTTP 200  
**And** el endpoint `GET /metrics` expone la métrica `model_version_info` con el label de la versión cargada  

### Escenario 4: Trazabilidad completa en W&B

**Given** que la promoción ha ocurrido  
**When** se accede al W&B Model Registry del proyecto  
**Then** se puede visualizar el historial de versiones de cada modelo  
**And** cada versión tiene asociado su run de entrenamiento con todas las métricas  
**And** se puede comparar side-by-side las métricas de v1 y v2 en la UI de W&B

## Definición de Done

- [ ] `compare_and_promote.py` recupera los runs más recientes de v1 y v2 desde la API de W&B
- [ ] La comparación se basa en RMSE sobre el conjunto de validación
- [ ] El umbral de promoción (5%) está definido como constante configurable via variable de entorno `PROMOTION_THRESHOLD`
- [ ] El script promueve exactamente un modelo a estado `production` en cada ejecución
- [ ] El script tiene código de salida 0 si la promoción fue exitosa, distinto de 0 si falló
- [ ] El GitHub Actions workflow ejecuta `compare_and_promote.py` después de ambos entrenamientos
- [ ] El servicio `ml-inference` usa la API de W&B para descargar el modelo en estado `production` al arrancar
- [ ] El endpoint `/ready` retorna 503 si el modelo no ha sido descargado aún
- [ ] Tests unitarios validan la lógica de comparación con mocks de la API de W&B
- [ ] El README documenta cómo visualizar la comparación de experimentos en W&B
