## Historia de Usuario

**Como** ingeniero de plataforma responsable del sistema,
**quiero** que cada paso del flujo de una consulta del usuario quede registrado como un evento en Apache Kafka, incluyendo la dimensión temporal completa (mes y año) y el resultado de las validaciones de rango,
**para** tener trazabilidad completa de extremo a extremo de cada interacción, auditar el comportamiento del sistema, calcular latencias por etapa y detectar patrones en las fechas solicitadas por los usuarios.

---

## Descripción

Cada servicio del sistema publica un evento en su topic de Kafka correspondiente cuando procesa una consulta. Los tres topics actúan como log de auditoría distribuido: `user-requests` registra la entrada del usuario incluyendo el año extraído cuando aplica, `agent-actions` registra el razonamiento del agente y el resultado de la validación temporal, y `model-responses` registra la predicción generada. Las validaciones temporales fallidas (fecha histórica o fuera del horizonte) también se registran en `agent-actions` para análisis de patrones de uso. Un servicio consumidor correlaciona eventos por `session_id` y calcula métricas de latencia y distribución de fechas consultadas.

---

## Esquemas de eventos por topic

### Topic: `user-requests`
```json
{
  "timestamp": "<ISO8601>",
  "session_id": "<uuid>",
  "pregunta": "<texto original del usuario>",
  "user_agent": "<browser info>"
}
```

### Topic: `agent-actions`
```json
{
  "timestamp": "<ISO8601>",
  "session_id": "<uuid>",
  "pregunta_original": "<texto>",
  "provincia_extraida": "<PROVINCIA | null>",
  "mes_extraido": "<1-12 | null>",
  "año_extraido": "<2025-2026 | null>",
  "validacion_temporal": {
    "es_valida": "<true | false>",
    "motivo_rechazo": "<fecha_historica | fuera_de_horizonte | null>",
    "fecha_limite_inferior": "2025-10",
    "fecha_limite_superior": "<calculada dinamicamente>",
    "meses_hacia_futuro": "<entero | null>"
  },
  "endpoint_construido": "<POST /predict | null>",
  "payload_enviado": "<objeto | null>",
  "latencia_agente_ms": "<float>"
}
```

### Topic: `model-responses`
```json
{
  "timestamp": "<ISO8601>",
  "session_id": "<uuid>",
  "provincia": "<PROVINCIA>",
  "mes": "<1-12>",
  "año": "<2025-2026>",
  "prediccion": "<float>",
  "modelo_version": "<v1 | v2>",
  "confianza": "<0.0-1.0>",
  "latencia_ms": "<float>"
}
```

## Criterios de Aceptación

### Escenario 1: Evento publicado en `user-requests` al recibir consulta

**Given** que el servicio `web-ui` está corriendo y conectado a Kafka
**When** el usuario envía cualquier pregunta a través del chat (válida o inválida temporalmente)
**Then** se publica inmediatamente un mensaje en el topic `user-requests` con `session_id`, `timestamp` y `pregunta`
**And** el mensaje se publica antes de llamar al servicio del agente
**And** el evento se registra independientemente de si la validación temporal posterior tiene éxito o falla

### Escenario 2: Evento publicado en `agent-actions` para consultas con fecha válida

**Given** que el agente procesa una consulta con provincia, mes y año dentro del rango permitido
**When** el agente completa su ciclo ReAct y obtiene la predicción
**Then** se publica un mensaje en `agent-actions` con todos los campos del esquema completos
**And** `validacion_temporal.es_valida` es `true`
**And** `validacion_temporal.motivo_rechazo` es `null`
**And** `validacion_temporal.meses_hacia_futuro` contiene el número de meses desde octubre 2025 hasta la fecha solicitada
**And** `endpoint_construido` es `"POST /predict"` y `payload_enviado` contiene el payload completo incluyendo `año`

### Escenario 3: Evento publicado en `agent-actions` para consultas con fecha inválida

**Given** que el agente recibe una consulta cuya fecha no supera la validación temporal
**When** el agente determina que la fecha es histórica o supera el horizonte máximo
**Then** se publica un mensaje en `agent-actions` con `validacion_temporal.es_valida` igual a `false`
**And** `validacion_temporal.motivo_rechazo` indica `"fecha_historica"` o `"fuera_de_horizonte"` según corresponda
**And** `endpoint_construido` y `payload_enviado` son `null` (no se llamó al modelo)
**And** `latencia_agente_ms` registra el tiempo total del ciclo incluyendo la validación

### Escenario 4: Evento publicado en `model-responses` incluye año

**Given** que el servicio `ml-inference` recibe y procesa una solicitud de predicción válida
**When** el modelo genera la predicción exitosamente
**Then** se publica un mensaje en `model-responses` con el campo `año` indicando el año para el que se predijo
**And** los campos `mes` y `año` juntos identifican unívocamente el período de predicción

### Escenario 5: Fallo de Kafka no interrumpe el flujo principal

**Given** que el broker de Kafka no está disponible
**When** cualquier servicio intenta publicar un evento
**Then** el servicio captura la excepción y registra WARNING: `[KAFKA] No se pudo publicar en topic <topic>: <error>`
**And** el flujo principal continúa sin interrupción
**And** el usuario recibe su respuesta normalmente
**And** la métrica Prometheus `kafka_publish_errors_total` se incrementa

### Escenario 6: El consumidor calcula métricas de distribución temporal

**Given** que el `kafka-consumer` procesa mensajes del topic `agent-actions`
**When** acumula eventos de consultas válidas e inválidas
**Then** calcula y expone como métricas Prometheus:
- `requests_by_provincia` (counter, label: provincia)
- `requests_by_año` (counter, label: año)
- `validation_failures_total` (counter, label: motivo_rechazo)
- `pipeline_end_to_end_latency_seconds` (histogram, calculada por session_id)

### Escenario 7: El consumidor correlaciona eventos por session_id

**Given** que los tres topics contienen eventos con el mismo `session_id`
**When** el `kafka-consumer` procesa los mensajes
**Then** calcula la latencia end-to-end como diferencia entre timestamp de `model-responses` y `user-requests` para ese `session_id`
**And** para sesiones con validación fallida (sin evento en `model-responses`), calcula la latencia entre `user-requests` y `agent-actions`

## Definición de Done

- [ ] Los tres topics (`user-requests`, `agent-actions`, `model-responses`) se crean automáticamente al iniciar Kafka
- [ ] Los esquemas JSON de cada topic están documentados en el README y en esta HU
- [ ] El campo `session_id` se genera en `web-ui` como UUID y se propaga a todos los servicios
- [ ] El campo `año_extraido` está presente en los eventos de `agent-actions` cuando el agente extrae un año de la consulta
- [ ] El bloque `validacion_temporal` está completo en todos los eventos de `agent-actions`, tanto para validaciones exitosas como fallidas
- [ ] El campo `año` está presente en los eventos de `model-responses`
- [ ] Cada servicio tiene su propio `kafka_producer.py` con manejo de errores fail-safe
- [ ] El `kafka-consumer` expone las métricas: `requests_by_provincia`, `requests_by_año`, `validation_failures_total` y `pipeline_end_to_end_latency_seconds`
- [ ] Los mensajes de error de Kafka aparecen como WARNING en los logs, nunca como ERROR crítico que interrumpa el flujo
- [ ] En `docker-compose.yml` los topics se crean con un servicio `kafka-init`
- [ ] Tests de integración validan que los eventos de validación fallida se publican con el esquema correcto
- [ ] El dashboard de Grafana incluye panel de `validation_failures_total` y `requests_by_año`
