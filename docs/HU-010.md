# HU-010: Resiliencia ante Fallos del Pipeline

## Identificación

| Campo | Detalle |
|---|---|
| **ID** | HU-010 |
| **Épica** | Infraestructura y Orquestación |
| **Prioridad** | Media |
| **Estimación** | 3 puntos |
| **Sprint** | 3 |

---

## Historia de Usuario

**Como** usuario del sistema,  
**quiero** que el sistema responda de forma degradada pero funcional cuando alguno de sus componentes internos no está disponible,  
**para** no encontrarme con errores técnicos incomprensibles ni pantallas en blanco cuando hay un fallo parcial en la infraestructura, y poder seguir usando el sistema en la medida de lo posible.

---

## Descripción

El sistema debe implementar patrones de resiliencia en cada punto de integración entre servicios: si Kafka no está disponible, el flujo continúa sin publicar eventos; si el servicio de inferencia no responde, el agente retorna un mensaje de error amigable; si el modelo no está cargado aún en el pod de inferencia, el endpoint `/ready` bloquea el tráfico hasta que esté listo. Cada fallo debe quedar registrado en los logs con suficiente contexto para diagnóstico, y las métricas de Prometheus deben reflejar la tasa de errores para visibilización en Grafana.

---

## Criterios de Aceptación

### Escenario 1: Kafka no disponible — el flujo principal continúa

**Given** que el broker de Kafka está caído o no accesible  
**When** cualquier servicio intenta publicar un evento  
**Then** el servicio captura la excepción de conexión  
**And** registra el error en el log con nivel WARNING: `[KAFKA] No se pudo publicar evento en topic <topic>: <error>`  
**And** el flujo principal del servicio continúa sin interrupción  
**And** el usuario recibe su respuesta normalmente  
**And** la métrica Prometheus `kafka_publish_errors_total` se incrementa  

### Escenario 2: Servicio de inferencia no disponible — respuesta degradada

**Given** que el pod de `ml-inference` está caído o no responde  
**When** el agente intenta llamar al endpoint `POST /predict`  
**Then** el agente captura el error de conexión después de un timeout de 5 segundos  
**And** retorna al `web-ui` un mensaje de error estructurado:
```json
{
  "respuesta": "Lo siento, el servicio de predicción no está disponible en este momento. Por favor intenta nuevamente en unos minutos.",
  "error": true,
  "detalle": "inference_service_unavailable"
}
```
**And** el `web-ui` muestra el mensaje de error al usuario en el chat con formato visual diferenciado  
**And** no se muestra ningún stack trace o mensaje técnico al usuario  

### Escenario 3: Modelo no cargado — readinessProbe bloquea el tráfico

**Given** que el pod de `ml-inference` está iniciando y el modelo aún no ha sido descargado desde W&B  
**When** Kubernetes evalúa la `readinessProbe` llamando a `GET /ready`  
**Then** el endpoint retorna HTTP 503 con body `{"status": "not_ready", "reason": "model_not_loaded"}`  
**And** Kubernetes no enruta ninguna petición a ese pod  
**And** el log del pod indica: `[MODEL] Modelo no disponible, esperando descarga desde W&B`  
**And** una vez descargado y cargado el modelo, `/ready` retorna HTTP 200 y el tráfico es enrutado  

### Escenario 4: El agente LLM no responde o supera el timeout

**Given** que la API de Groq no está disponible o responde con latencia excesiva  
**When** el agente intenta hacer una llamada a LiteLLM  
**Then** la llamada tiene un timeout configurado de 30 segundos  
**And** si el timeout es superado, el agente registra el error con nivel ERROR  
**And** retorna una respuesta degradada indicando que el asistente no está disponible  
**And** la métrica `agent_llm_errors_total` se incrementa  

### Escenario 5: Los errores son visibles en el dashboard de Grafana

**Given** que han ocurrido fallos en alguno de los servicios  
**When** se visualiza el dashboard de Grafana  
**Then** el panel "Tasa de errores por servicio" muestra un incremento en la línea del servicio afectado  
**And** el panel "Kafka publish errors" muestra los fallos de publicación si los hubo  
**And** el panel "Agent LLM errors" muestra los fallos de llamada al LLM si los hubo

## Definición de Done

- [ ] Cada `kafka_producer.py` tiene bloque try/except con logging WARNING y nunca relanza la excepción
- [ ] El agente tiene timeout de 5 segundos configurado en la llamada al servicio de inferencia
- [ ] El agente tiene timeout de 30 segundos configurado en la llamada a LiteLLM
- [ ] Los mensajes de error retornados al usuario están en español y son comprensibles
- [ ] El `web-ui` diferencia visualmente los mensajes de error de los mensajes normales en el chat
- [ ] El endpoint `/ready` de `ml-inference` retorna 503 mientras el modelo no está cargado
- [ ] Las métricas `kafka_publish_errors_total` y `agent_llm_errors_total` están implementadas
- [ ] El panel de tasa de errores está incluido en el dashboard de Grafana
- [ ] Tests unitarios validan el comportamiento fail-safe del kafka_producer con mocks
- [ ] Tests unitarios validan que el agente retorna respuesta degradada cuando inferencia falla
